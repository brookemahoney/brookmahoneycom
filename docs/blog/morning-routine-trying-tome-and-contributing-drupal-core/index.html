Morning routine, Trying Tome, and Contributing to Drupal Core

Morning, ya'll

When it's morning, I catch up on reading. I check drupal planet, ycombinator, youtube subs, medium digest, email, this, that, and the other. If every day I do this, I know I'm still developing at the cutting edge. When something new comes out, I will always at least see the headlines.

Yesterday was productive. I learned that Tome is a cool thing for static site generators for Drupal and that it integrates with Gatsby or anything else that can digest JSON. This seems to be a great thing to use. Even if it's buggy, it will be fine, because I can help fix it.

I like the idea of Tome, which can use Drupal's caching and rendering pipelines. Does the Drupal Gatsby module do anything similar or does it just hook up to GraphQL? What does the module ecosystem look like for Drupal Gatsby versus Tome? Does it makes sense for the two modules to work together so that the Gatsby module provides a generator entity type for Tome?

I think the first thing I'll do is spin up a default drupal site and then add Tome to it and see what that does.

Spinning up a Tome Drupal site locally

FirstStart Orbstack. Then, Go to the DDEV docs for the most up to date instructions on creating a new Drupal site: https://docs.ddev.com/en/stable/users/quickstart/#drupal. At the time of this writing, this is how it's done:

mkdir tometest && cd tometest
ddev config --project-type=drupal11 --docroot=web
ddev start
ddev composer create-project "drupal/recommended-project:^11"
ddev composer require drush/drush
ddev drush site:install --account-name=admin --account-pass=admin -y
ddev launch
# or automatically log in with
ddev launch $(ddev drush uli)
 

Next, I'll visit https://tome.fyi/docs/installing-existing-site/ and follow the instructions there to install Tome on an existing site.

ddev composer require drupal/tome
ddev drush en tome_static -y
ddev drush tome:static --uri=http://localhost:8001
cd html
python -m http.server 8001
open http://localhost:8001 

OMG, are you kidding me? It worked, first time! The Drupal home page is a static page in the /html folder of the install. I mean, this seems amazing.

The road to Drupal Core contributions.

I've tried out Drupal CMS. When I started to dig into what the new core modules are. I see there's an Announcements module but I can't tell what it does. I install the help module and see that the help page for it could use a bit of improvement. This is a perfect time for me to set up my computer for Drupal Core development. After a lot of time searching through various methods, I came across the DDEV concept of "add-ons". There are add-ons for contributing to Drupal core and contrib. Woo hoo.
https://addons.ddev.com/addons/justafish/ddev-drupal-core-dev
https://addons.ddev.com/addons/ddev/ddev-drupal-contrib

OK, the new plan (versus the old way of making a diff and a patch file) to contribute to Drupal core is:

  • Use ddev to create an environment to test changes using the ddev core add-on
  • Once satisfied, make an issue on d.o.
  • When the issue is created, a button appears that allows for creating an issue fork.
  • Using a separate directory from the one you test locally in, commit your changes to that branch and push back up.
  • For testing it on the web, consider using DrupalPod (See https://www.drupaleasy.com/blogs/ultimike/2023/12/using-drupalpod-core-and-contrib-development)

Drupal contrib

When I was looking at modules in Drupal CMS, I noticed Easy Breadcrumb in there. Well, lo and behold, I'm one of the maintainers of that module. Cool! I was reading the help file for it and noticed that a link needed to be updated. So, I made an issue, used the Create Issue Fork button on the issue page, and used the commands there to help me commit my changes and push them back up. When I did that, the terminal gave me a link in Gitlab to create my merge request. I visited the link, filled out the form, and it was done, my first ever merge request for a Drupal contrib project hosted on Gitlab. After that, I went back to the issue, marked it as Needs Review, and unassigned it from myself. As a maintainer, I guess I can go merge the code, but that seems wrong. Hopefully Greg will see the request and merge it in soon.
